Find the writer, run the writer

Views-vs-DB discipline

Editing a generated JSON view by hand is editing a copy — the change vanishes on the next regeneration, or worse, desyncs the view from the DB. Every registry has a designated writer; you change state by running it.

Hand-editing is editing a copy

If the JSON files are generated from the DB, then editing a JSON view by hand is editing a copy — your change is real until the next regeneration, then it silently vanishes. Or worse: it desyncs the view from the DB and misleads every reader until someone notices. So the discipline is absolute: do not hand-edit a generated view. TRACKING/registries.json literally annotates the generated artifacts with "Do NOT hand-edit."

Every registry has a writer

You change state by invoking the registry's designated writer, which writes the DB and re-exports the view. Examples named in the registry itself:

  • dispatches — written by scripts/update-dispatches-registry.py, invoked from /accept-session-dispatch. Never edit dispatches.json directly.
  • kaizen — written by /kaizen-log, re-exported by scripts/legacy-export.py.
  • stubs / workorders / tickets — minted and transitioned through their skills (/stub, /create-workorder, /ticket), which use the canonical DB-insert path.

Hand-editing also corrupts

Beyond desync, hand-editing JSON is mechanically corruption-prone. The dispatch runners warn about exactly this (known issue KI-004): hand-appending fields to a dispatch JSON drops commas and braces and breaks the file. The runner round-trips safely via JSON.stringify; a human editor does not.

The rule of thumb

Find the writer, run the writer. If you can't find a writer for a change you want to make, that's a signal the change is wrong — not that hand-editing is okay.

Canonical sources

  • TRACKING/registries.json — names the writer for each registry.
  • SKILLS/SESSION/accept-session-dispatch/SKILL.md — the safe round-trip contract for dispatch state.
