Never freehand what a skill owns

Finding and creating skills

Two surfaces find a skill — the generated SKILL-INDEX table and the /which-skill ranked search. One surface creates them — /new-coderef-skill. The standing rule: never freehand a workflow a canonical skill already owns.

Finding a skill

Two surfaces:

  • SKILLS/STANDARDS/SKILL-INDEX.md — the generated lookup-by-purpose table: one row per skill with its slash command, summary, and path. It is regenerated on every registry rebuild, so it is always current with the corpus on disk.
  • /which-skill — the ranked search over that index, for when you have a topic but not a name. Ask it "what handles X" and it returns the top candidates with category and a one-line summary.

The standing rule

Never freehand a workflow that a canonical skill already owns. If a skill exists for the job, the skill is the job. Freehanding past it forks the workflow: the canonical contract keeps improving while the freehand copy silently rots — and the next agent inherits the divergence.

Creating a skill

The same rule covers authorship: never create or modify a skill except through /new-coderef-skill, the authoring skill. It scaffolds conformant packages, keeps SKILL.md and skill.json versions in lockstep, installs the thin wrappers into each CLI surface, and rebuilds the registry — all the invariants the validators check, satisfied by construction.

Canonical sources

  • SKILLS/STANDARDS/SKILL-INDEX.md — the lookup table.
  • SKILLS/WORKFLOW/new-coderef-skill/SKILL.md — the authoring skill.
