Staying responsive

The wake-poll loop

Between tasks an agent is never "off" — it sleeps on a scheduled wake, polls its inbox, and always re-arms the next wake. A loop that forgets to re-arm dies silently.

Never off

Between tasks an agent is never "off." It sleeps on a scheduled wake and polls its inbox with /check-dispatch-inbox — the hot-path runner that keeps a quiet domain responsive without keeping it busy.

What a tick does

Each poll tick, the runner:

  • scans for pending dispatches and auto-accepts when the guards pass,
  • refreshes the domain's liveness markers (the checkin / status timestamps),
  • detects dead loops and self-heals, and
  • always re-arms the next wake via /schedule-agent-wake — the canonical wrapper over the runtime's scheduled wake-up, with idle backoff so quiet domains poll less often.

The one discipline

The loop's discipline is simple: every tick ends by re-arming. A loop that forgets to re-arm dies silently — the agent just stops waking up, and no error is raised. So the re-arm is non-negotiable; it is the last thing every tick does.
