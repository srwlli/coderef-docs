How work moves

The dispatch flow

Dispatches are how work moves between domains — a sender files one, the receiver accepts and completes it through a state-machine runner, and the result is written back onto the same artifact.

Dispatches are how work moves between domains.

Filing a dispatch

A sender files a dispatch with /dispatch-session-request — an atomic write of a dispatch-NNN.json into the receiver's session directory, plus a session event. The dispatch carries the task, the objective, and a priority.

The receiver's lifecycle

The receiver's lifecycle runs through /accept-session-dispatch — a state-machine runner that moves a dispatch pending → accepted → completed, with blocked as the escalation lane. It:

  • round-trips the JSON safely — hand-editing a dispatch file is the classic corruption bug, so the runner owns the writes,
  • enforces skill gates when a dispatch requires specific skills to have been logged, and
  • on completion, writes the result summary back onto the dispatch itself, so the sender reads the outcome from the same artifact it created.

Status follows the lifecycle

The domain's status truth follows the lifecycle automatically: accepting a dispatch marks the domain active on that task; completing it returns the domain to idle. No separate status bookkeeping — the dispatch state is the source.
