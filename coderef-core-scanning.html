Scanning and artifacts

How a project gets its .coderef/ artifacts — bootstrap with coderef-init, generate with populate-coderef, read quickly with coderef-scan, repair with coderef-pipeline, and keep current with coderef-watch.

Bootstrap a project

/coderef-init bootstraps a project for the engine: it creates the canonical coderef/ and .coderef/ directory structures plus a starter .coderefignore. Run it once per project; everything else builds on the layout it creates.

Generate and refresh

/populate-coderef is the single-stage workhorse: it generates or refreshes the .coderef/ artifacts — the element-level index, the dependency graph, and the report files — for a target project. When artifacts are missing or stale, this is the skill that rebuilds them.

Quick reads

/coderef-scan gives element-level stats without writing artifacts — the quick "what's in here" read when you want a picture of a codebase but don't need to persist anything.

The pipeline

/coderef-pipeline is the multi-stage coordinator: scan, then populate, then validate, in one run. This is what workorder planning invokes to repair missing or drifted artifacts — one command that takes a project from unknown state to verified artifacts.

Staying current

/coderef-watch runs a long-lived watcher over a project's source and its .coderef/ directory, emitting change notifications so artifacts stay current instead of silently drifting from the code they describe.

Canonical sources

  • CODEREF-CORE/docs/CLI.md — the authoritative command reference.
  • SKILLS/CORE/coderef-init/SKILL.md, SKILLS/CORE/coderef-pipeline/SKILL.md — the skill contracts wrapping these verbs.
