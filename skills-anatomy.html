One directory, one contract

Anatomy of a skill package

Every skill is a directory at SKILLS/{CATEGORY}/{skill-name}/ with a standard shape — SKILL.md as the agent-facing contract, skill.json as the machine manifest, a runner when there is deterministic work, and support folders only when earned.

The package shape

Every skill is a directory at SKILLS/{CATEGORY}/{skill-name}/ with a standard anatomy. Four pieces matter; each has one job.

SKILL.md — the contract

SKILL.md is the canonical agent-facing contract: frontmatter (name, description, metadata.version), ## Instructions between START/STOP markers, and ## Common Issues.

The description is a signpost — what the skill does and when to use it, with no changelog history. History lives in references/CHANGELOG.md, and a validator gate refuses descriptions that drift into changelog territory.

skill.json — the manifest

skill.json is the machine-readable manifest: id, version (kept in lockstep with SKILL.md), category, summary, and the reads/writes/depends_on contract. Discovery, validation, and tracking all consume this file — and TRACKING/skills.json is rebuilt from the manifests on disk, never hand-edited.

run.mjs — the runner

When a skill has deterministic work, it ships run.mjs — the runner. Every runner answers --help via a shared emitter, so any agent can interrogate any runner the same way. Prose-only skills are exempt: no mechanics, no runner.

Support folders, when earned

Support folders exist only when the skill has earned them:

  • references/ — stable depth, including the changelog.
  • assets/ — templates the skill stamps out.
  • scripts/ — validators and helpers.
  • standards.json — behavior-seed pointers.
  • kaizen.md — the per-skill continuous-improvement sidecar every run can append to.

Canonical sources

  • SKILLS/WORKFLOW/new-coderef-skill/SKILL.md — the authoring skill, plus its references/ for the packaging and wrapper standards.
  • SKILLS/STANDARDS/SKILL-CONTRACTS.md — what "MANDATORY" language must be backed by.
