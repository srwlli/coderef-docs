The DB is the truth

The registry-db model

DB truth, JSON views

CodeRef tracks its work in a small SQLite database (registry.db). That database is the single source of truth. The many *.json files under TRACKING/ that agents and tools read every day are generated views — projections of the database, regenerated by export scripts. They exist because JSON is cheap to read, diff, and grep; the DB is where writes land.

The governing rule

When a view and the DB disagree, the DB wins.

A view can be stale (it hasn't been regenerated since the last write) or locally dirty (someone edited the file by hand). Neither changes the truth — the DB does. The fix for a wrong-looking view is always regenerate it from the DB, never patch the JSON.

See it in the registry

You can see the model directly in TRACKING/registries.json. Entries carry a type:

  • "type": "view" with a "source" (e.g. unified-registry.json or a registry.db table) and a "view_filter" — a generated projection. Examples: workorders, stubs, skills, sessions, dispatches, kaizen.
  • "type": "registry" / "json-index" — a directly-maintained artifact with a named owner.

The kaizen registry states it plainly: "Source-of-truth is the DB; this JSON is regenerated by scripts/legacy-export.py." That sentence is the whole philosophy in one line.
